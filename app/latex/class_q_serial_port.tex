\doxysubsection{Detailed Description}
Provides functions to access serial ports. 

\textbackslash{}reentrant

\textbackslash{}inmodule Qt\+Serial\+Port \begin{DoxySince}{Since}
5.\+1
\end{DoxySince}
You can get information about the available serial ports using the \mbox{\hyperlink{class_q_serial_port_info}{Q\+Serial\+Port\+Info}} helper class, which allows an enumeration of all the serial ports in the system. This is useful to obtain the correct name of the serial port you want to use. You can pass an object of the helper class as an argument to the \mbox{\hyperlink{class_q_serial_port_a595e5a5cfd5cd0f7f53e35b439c7a6cf}{set\+Port()}} or \mbox{\hyperlink{class_q_serial_port_a34a8477c8deda835d8d9986f38834b0f}{set\+Port\+Name()}} methods to assign the desired serial device.

After setting the port, you can open it in read-\/only (r/o), write-\/only (w/o), or read-\/write (r/w) mode using the \mbox{\hyperlink{class_q_serial_port_a2d6586b11209863aa05574975f79615f}{open()}} method.

\begin{DoxyNote}{Note}
The serial port is always opened with exclusive access (that is, no other process or thread can access an already opened serial port).
\end{DoxyNote}
Use the \mbox{\hyperlink{class_q_serial_port_a2e221672652676a33daaf7f1a2228294}{close()}} method to close the port and cancel the I/O operations.

Having successfully opened, \mbox{\hyperlink{class_q_serial_port}{Q\+Serial\+Port}} tries to determine the current configuration of the port and initializes itself. You can reconfigure the port to the desired setting using the set\+Baud\+Rate(), set\+Data\+Bits(), set\+Parity(), set\+Stop\+Bits(), and set\+Flow\+Control() methods.

There are a couple of properties to work with the pinout signals namely\+: \mbox{\hyperlink{class_q_serial_port_a90d9cb53f0b7948a1e2166fd395e7458}{Q\+Serial\+Port\+::data\+Terminal\+Ready}}, \mbox{\hyperlink{class_q_serial_port_a918269d1550b4fe0d1333cdee98e8020}{Q\+Serial\+Port\+::request\+To\+Send}}. It is also possible to use the \mbox{\hyperlink{class_q_serial_port_a8f88f140bf533b8f97229163209c585f}{pinout\+Signals()}} method to query the current pinout signals set.

Once you know that the ports are ready to read or write, you can use the read() or write() methods. Alternatively the read\+Line() and read\+All() convenience methods can also be invoked. If not all the data is read at once, the remaining data will be available for later as new incoming data is appended to the \mbox{\hyperlink{class_q_serial_port}{Q\+Serial\+Port}}\textquotesingle{}s internal read buffer. You can limit the size of the read buffer using \mbox{\hyperlink{class_q_serial_port_affb5534880671d30438b2bc07b0008b1}{set\+Read\+Buffer\+Size()}}.

\mbox{\hyperlink{class_q_serial_port}{Q\+Serial\+Port}} provides a set of functions that suspend the calling thread until certain signals are emitted. These functions can be used to implement blocking serial ports\+:

\textbackslash{}list

\begin{DoxyItemize}
\item \mbox{\hyperlink{class_q_serial_port_a5a274fca3c9315fb5536cc8e716e4d9a}{wait\+For\+Ready\+Read()}} blocks calls until new data is available for reading.\end{DoxyItemize}
\begin{DoxyItemize}
\item \mbox{\hyperlink{class_q_serial_port_a4df1aac683ab204adc11eaafc4c1cc0b}{wait\+For\+Bytes\+Written()}} blocks calls until one payload of data has been written to the serial port.\end{DoxyItemize}
\textbackslash{}endlist

If \textbackslash{}l\{Q\+I\+O\+Device\+:\+:\}\{\mbox{\hyperlink{class_q_serial_port_a5a274fca3c9315fb5536cc8e716e4d9a}{wait\+For\+Ready\+Read()}}\} returns {\ttfamily false}, the connection has been closed or an error has occurred.

If an error occurs at any point in time, \mbox{\hyperlink{class_q_serial_port}{Q\+Serial\+Port}} will emit the \mbox{\hyperlink{class_q_serial_port_a5614a5d76859606e2c2aed87da333157}{error\+Occurred()}} signal. You can also call \mbox{\hyperlink{class_q_serial_port_ad3b0d74404cf2d5432552d4bf578058a}{error()}} to find the type of error that occurred last.

\begin{DoxyNote}{Note}
Not all error conditions are handled in a platform independent way in Q\+Serialport, as for example the Framing, Parity, and Break condition errors. These kind of errors need to be handled by the application code, probably using OS system specific ioctls on the device descriptor and/or parsing the stream\textquotesingle{}s byte-\/stuffing.
\end{DoxyNote}
Programming with a blocking serial port is radically different from programming with a non-\/blocking serial port. A blocking serial port does not require an event loop and typically leads to simpler code. However, in a G\+UI application, blocking serial port should only be used in non-\/\+G\+UI threads, to avoid freezing the user interface.

For more details about these approaches, refer to the \textbackslash{}l \{Qt Serial Port Examples\}\{example\} applications.

The \mbox{\hyperlink{class_q_serial_port}{Q\+Serial\+Port}} class can also be used with Q\+Text\+Stream and Q\+Data\+Stream\textquotesingle{}s stream operators (operator$<$$<$() and operator$>$$>$()). There is one issue to be aware of, though\+: make sure that enough data is available before attempting to read by using the operator$>$$>$() overloaded operator.

In most cases, this function does not need to be called, because the \mbox{\hyperlink{class_q_serial_port}{Q\+Serial\+Port}} class will start sending data automatically once control is returned to the event loop. In the absence of an event loop, call \mbox{\hyperlink{class_q_serial_port_a4df1aac683ab204adc11eaafc4c1cc0b}{wait\+For\+Bytes\+Written()}} instead.

Sets the size of \mbox{\hyperlink{class_q_serial_port}{Q\+Serial\+Port}}\textquotesingle{}s internal read buffer to be {\itshape size} bytes.

If the buffer size is limited to a certain size, \mbox{\hyperlink{class_q_serial_port}{Q\+Serial\+Port}} will not buffer more than this size of data. The special case of a buffer size of {\ttfamily 0} means that the read buffer is unlimited and all incoming data is buffered. This is the default.

This option is useful if the data is only read at certain points in time (for instance in a real-\/time streaming application) or if the serial port should be protected against receiving too much data, which may eventually cause the application to run out of memory.
